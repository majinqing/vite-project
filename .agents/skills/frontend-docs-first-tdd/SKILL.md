执行前端“文档先行 + 严格 TDD”流程。

先读取以下参考文件，再开始工作：

* `references/workflow.md`：完整流程、拦截条件、例外处理和汇报格式。

* `references/doc-templates.md`：需求文档、测试用例文档和结果汇报模板。

* `references/repo-adapter-zsjd-data-portal-web.md`：当前仓库专属约束与测试入口。

## 核心原则

* 先识别任务类型，再决定是否进入严格 TDD。

* 只要涉及前端行为变化，默认进入严格 TDD。

* 没有需求确认结论，不进入实现。

* 没有需求文档和测试用例文档，不进入 TDD。

* 没有失败测试，不进入实现阶段。

* 需求变化时，先更新两份文档，再改测试和代码。

* 优先复用现有工具、组件和测试辅助能力，不重复造轮子。

## 触发后的默认动作

1. 先复述你对任务的理解。

2. 明确当前任务类型，以及是否适用严格 TDD。

3. 先检查仓库规则、现有工具、现有测试和现有文档样例。

4. 产出需求确认结果。

5. 明确两份文档路径与目标测试文件路径。

6. 先写失败测试，再进入最小实现和重构。

7. 最后按固定格式汇报文档、测试、实现、验证、偏差。

## 何时允许降级

仅在以下情况允许偏离完整流程，并且必须先说明原因：

* 纯文案调整，且不改变行为。

* 纯样式调整，且不改变行为。

* 极小修复，确实不值得完整需求文档，但仍需保留测试说明。

* 测试基础设施不足，只能执行最小可行验证。

## 行为要求

* 对缺失信息先做本地探索，不要第一时间追问用户。

* 遇到流程前置条件不满足时，明确拦截，并指出缺哪一步。

* 组件测试优先验证用户可见行为，不鼓励围绕内部实现写脆弱测试。

* 汇报时优先说明是否完成“先失败，再通过”。

* 如果无法执行测试，必须写明阻塞原因、已做验证和剩余风险。
