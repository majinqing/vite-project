# 文档模板与命名规范

## 1. 文件命名规范

默认采用“一项需求对应两份文档”：

* 需求文档：`docs/<需求名称>.md`

* 测试用例文档：`docs/测试用例/<需求名称>测试用例.md`

命名要求：

* 默认使用中文文件名

* 名称直接体现需求或问题本身

* 不引入复杂编号，除非仓库已有明确编号体系

## 2. 需求文档模板

```md
# <需求名称>

## 需求背景

## 目标与价值

## 功能范围

## 用户操作流程

## 验收标准

## 影响范围

## 待确认项 / 风险点
```

最低要求：

* 即使是小需求，也要交代目标、范围和验收标准

* 如果是修复类任务，背景中写清触发条件和期望修复结果

## 3. 测试用例文档模板

```md
# <需求名称>测试用例

## 测试对象

## 前置条件

## 核心场景

## 异常与边界场景

## 回归检查点

## 用例步骤与预期结果
```

最低要求：

* 核心场景能直接映射到首轮失败测试

* 异常与边界场景不要只写“无”

* 回归检查点要覆盖最容易被带坏的旧行为

## 4. 测试计划模板

在进入 RED 前，先用下面的结构明确计划：

```md
## 测试计划

* 测试层级：

* 目标测试文件：

* 首轮失败测试：

* 用例映射：

* 复用能力：
```

说明：

* “测试层级”用于确定放到 `unit/stores/hooks/router/components` 哪一层

* “用例映射”用于说明每条测试对应哪条测试用例

* “复用能力”用于说明复用哪些 mock、helper、setup 或已有工具

## 5. 结果汇报模板

```md
## 文档

* 需求文档：

* 测试用例文档：

* 更新说明：

## 测试

* 测试文件：

* 覆盖场景：

* 是否先失败再通过：

## 实现

* 最小实现：

* 复用情况：

## 验证

* 执行命令：

* 执行结果：

## 偏差

* 流程偏差：

* 原因：

* 风险：
```

## 6. 例外场景记录模板

如果任务允许降级，至少记录：

```md
## 例外说明

* 降级原因：

* 为什么判断为无行为变化或极小修复：

* 最小验证方式：

* 未覆盖风险：
```
