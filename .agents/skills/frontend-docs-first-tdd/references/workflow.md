# 前端文档先行 + 严格 TDD 工作流

## 1. 适用范围

默认用于以下任务：

* 前端功能开发

* 行为修改

* 缺陷修复

* 重构

* 补单元测试、组件测试、路由测试、store 测试、hooks 测试

* 补需求文档或测试用例文档

默认不直接套完整流程的场景：

* 纯样式调整，且无行为变化

* 纯文案调整，且无行为变化

即使降级，也必须先说明为什么可以不走完整 TDD。

## 2. 首轮判断

进入任务后，先做四个判断：

1. 这次任务有没有前端行为变化。

2. 这次任务属于开发、修复、重构、补测，还是纯样式/纯文案。

3. 当前仓库是否已有对应工具、测试、文档模板或相似样例。

4. 当前测试基础设施是否足够支持先写失败测试。

如果第 1 条为“有”，默认进入严格 TDD。

## 3. 先探索，再确认

在提问前优先做本地探索，至少检查：

* 仓库规则文件，例如 `AGENTS.md`

* 工具类文档，例如 `src/utils/README.md`

* 测试命令与配置，例如 `package.json`、`vitest.config.ts`

* 现有测试目录与相似测试

* `docs/` 与 `docs/测试用例/` 现有样例

探索后先输出需求确认结果，至少包含：

* 背景

* 目标

* 范围

* 验收标准

* 风险 / 待确认项

如果仍有高影响歧义，再向用户确认。

## 4. 文档先行

在进入测试或实现前，先新增或更新两份文档：

* `docs/<需求名称>.md`

* `docs/测试用例/<需求名称>测试用例.md`

文档内容要求见 `doc-templates.md`。

若任务很小，可简化需求文档篇幅，但不能省略测试用例说明。

## 5. 测试计划

在写测试前，先明确：

* 测试层级：`test/unit`、`test/stores`、`test/hooks`、`test/router`、`test/components`

* 目标测试文件路径

* 首轮要写哪些失败测试

* 每条测试对应哪条测试用例

* 需要复用哪些现有 mock、helper 或 setup

优先使用已有测试辅助能力，不要重复搭建相同 mock。

## 6. Red -> Green -> Refactor

### RED

* 先编写能表达目标行为的测试

* 运行相关测试

* 确认测试先失败

如果测试一开始就通过，先检查：

* 现有行为是否已经覆盖该需求

* 测试断言是否失真

* 测试是否没有真正命中目标代码

### GREEN

* 只实现让当前测试通过所必需的最少代码

* 不顺手加额外能力

* 复跑相关测试，确认通过

### REFACTOR

* 只整理结构，不改变行为

* 允许改命名、提炼重复逻辑、收敛分支

* 重构后再次运行相关测试

## 7. 严格拦截规则

满足任一条件时，必须先拦截，不得直接实现：

* 没有需求确认结论

* 没有需求文档

* 没有测试用例文档

* 没有失败测试

* 需求已变化但文档未更新

拦截时要明确说清：

* 当前缺的前置条件是什么

* 下一步应先补什么

* 为什么现在不能直接进入实现

## 8. 例外与降级规则

### 纯文案或纯样式

可不走完整严格 TDD，但必须先说明：

* 为什么判断为“无行为变化”

* 为什么可不写完整需求文档

* 采用什么最小验证方式

### 极小修复

可以简化需求文档，但仍需：

* 记录修复目标

* 记录测试场景

* 尽量保留失败测试 -> 通过测试的链路

### 测试设施不足

不允许直接跳过说明。必须显式输出：

* 哪个环节无法跑

* 当前最小可行验证策略

* 尚未覆盖的风险

## 9. 默认输出格式

默认按以下五段汇报：

### 文档

* 需求文档路径

* 测试用例文档路径

* 本次新增或更新了什么

### 测试

* 目标测试文件路径

* 新增或修改了哪些测试场景

* 是否先失败再通过

### 实现

* 最小实现改了什么

* 是否有复用现有工具或 helper

### 验证

* 执行了哪些命令

* 结果如何

### 偏差

* 哪些地方偏离了标准流程

* 为什么偏离

* 剩余风险是什么
