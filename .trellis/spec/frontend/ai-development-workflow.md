# 前端 AI 辅助开发流程

> 适用于本仓库中的前端功能开发、前端行为修改、前端缺陷修复、影响行为的重构、补测试与相关验证工作。

---

## 目标

把“前端行为变化默认走 docs-first + TDD”固定为本仓库的 AI 协作约定，避免不同会话、不同 agent 对前端任务采取不一致的做法。

本流程在当前项目中的正式来源是：

- `AGENTS.md`
- `.trellis/spec/frontend/ai-development-workflow.md`
- `.trellis/workflow.md`

如果三者出现冲突，应立即修正文档，保持触发条件、降级规则和最低验证标准一致。

---

## 何时进入这条流程

以下情况默认进入 `frontend-docs-first-tdd`：

- 新增前端功能
- 修改现有交互或用户可见行为
- 修复前端缺陷
- 重构组件、页面、store、router、composable、utils，且可能影响行为
- 补前端单测、组件测试、交互验证

以下情况允许不走严格 TDD，但仍需保留验证说明：

- 纯问答、解释、规则说明
- 纯文案调整，且不改变行为
- 纯样式调整，且不改变行为
- 极小修复，且完整 TDD 成本明显高于收益
- 现有测试覆盖无法直接支撑，只能执行最小可行验证

---

## 与 Trellis 主流程的衔接

1. 需求不清或方案未定时，先走 `trellis-brainstorm`，把需求、范围、验收标准写入 `prd.md`。
2. 一旦确认任务存在前端行为变化，执行阶段先遵循 `frontend-docs-first-tdd`。
3. 实现完成后，仍按 Trellis 主流程继续执行 `trellis-check`、`trellis-update-spec`、commit 和 `/trellis:finish-work`。

这条流程是前端任务的实现约束，不替代 Trellis 的任务生命周期。

---

## 固定执行顺序

### 1. 先确认需求，再开始实现

- 没有需求确认结论，不进入实现。
- 需求变化时，先更新任务文档，再改测试和代码。
- 需求不明确时，优先通过仓库探索和 `trellis-brainstorm` 收敛，而不是直接开写。

### 2. 先补双文档，再进入 TDD

进入严格流程后，至少要把两类信息写清楚：

- 需求/范围：通常落在 `prd.md`
- 测试用例/验证计划：落在当前任务目录中的说明文档，确保后续实现和验证有可追踪依据

如果这两类信息还不完整，不进入实现阶段。

### 3. 严格按 `Red -> Green -> Refactor`

- 先写失败测试或失败验证用例
- 再做最小实现让验证通过
- 最后在保持行为正确的前提下重构

如果当前任务无法编写自动化测试，必须明确写出阻塞原因，并把“失败验证 -> 通过验证”的替代步骤记录下来。

---

## 降级规则

仅在以下场景允许降级，不执行完整 docs-first + 严格 TDD：

- 纯文案调整
- 纯样式调整
- 极小修复
- 现有测试覆盖无法支撑，且当前任务内不适合补齐

降级时必须满足：

- 先说明为什么降级
- 仍然保留需求确认结论
- 仍然保留测试说明或验证说明
- 不把“直接改代码”当作默认做法

---

## 当前仓库的最低验证标准

本仓库当前已经具备基础 `Vitest + Vue Test Utils` 能力，但测试覆盖并不完整，因此最低验证标准固定如下：

- 涉及实际前端代码改动时，至少执行 `npm.cmd run build`
- 能落入现有测试覆盖范围的行为，优先补或更新 `npm run test:unit`
- 涉及 UI 变化时，补充针对性的手工冒烟验证说明
- 涉及路由、表单、状态流、提交链路变化时，说明关键场景如何验证通过

如果无法完成上述验证，必须明确写出阻塞原因、已完成的验证和剩余风险。

## 流程验证办法

在当前项目中，验证这套流程是否真正落地，固定按以下 4 组检查执行：

### 1. 文档静态验证

- 读取 `AGENTS.md`
- 读取 `.trellis/spec/frontend/index.md`
- 读取 `.trellis/spec/frontend/ai-development-workflow.md`
- 读取 `.trellis/workflow.md`

核对以下 3 组内容在语义上完全对齐：

- 哪些任务进入 `frontend-docs-first-tdd`
- 哪些场景允许降级
- 最低验证要求是什么

同时确认 `.trellis/spec/frontend/index.md` 已暴露 `ai-development-workflow.md`，让 `trellis-before-dev` 和 `trellis-check` 能从索引入口读到它。

### 2. 场景路由验证

用以下样例任务做人工演练，每个样例只能得到一个结论：

1. “新增一个筛选交互功能”
   预期：先 `trellis-brainstorm`，确认需求后进入 `frontend-docs-first-tdd`
2. “修复一个表单提交流程 bug”
   预期：直接判定为前端行为变化任务，进入 `frontend-docs-first-tdd`
3. “只改按钮文案”
   预期：允许降级，不走严格 TDD，但仍要求验证说明
4. “只调页面间距和颜色”
   预期：允许降级，不走严格 TDD，但仍要求验证说明

### 3. 执行链路验证

选择一个真实或模拟前端任务，检查整条链路是否闭环：

- 是否创建 Trellis task
- 若需求不清，是否先进入 `trellis-brainstorm`
- 是否在执行阶段被明确路由到 `frontend-docs-first-tdd`
- 是否产出需求/范围与测试/验证依据
- 是否在收尾时经过 `trellis-check`
- 是否在结果汇报里说明“先失败，再通过”或说明降级原因

### 4. 最低验证标准验证

对无法完整走自动化测试的前端任务，仍需核对：

- 是否执行 `npm.cmd run build`
- 若行为能落入现有测试覆盖，是否执行 `npm run test:unit`
- 是否补充了手工冒烟验证说明
- 是否说明剩余风险

只要以上 4 组检查都能通过，就可以认为这套流程在当前项目中已完成落地并具备可用性。

---

## AI 执行注意事项

- 先探索仓库现有实现、现有测试、现有文档，再提问用户。
- 优先验证用户可见行为，不鼓励围绕内部实现写脆弱测试。
- 完成汇报时，要明确说明这次是否满足“先失败，再通过”。
- 如果本次任务产出了新的前端协作约定或踩坑经验，应回写到 `.trellis/spec/frontend/`。
