# 质量规范

> 这个项目的前端代码质量标准。

---

## 总览

当前仓库的质量检查主要围绕 TypeScript、Vite 构建、ESLint，以及是否保持 Vue 3 项目约定展开。仓库已经具备基础 `Vitest + Vue Test Utils` 能力，但不是所有前端行为都已有完整自动化覆盖，因此构建、测试与针对性验证需要一起看。

---

## 禁止模式

- 不要在新代码里重新引入 Options API。
- 不要在改动文件时把 `@` 别名改成长层级相对路径。
- 不要新增无类型 props、`any`，或可以避免的非空断言。
- 不要偷偷修改路由名、路径或 `meta` 契约而不更新依赖它们的菜单和布局逻辑。
- 不要想当然地依赖自动导入，先确认当前 resolver 已覆盖该 API。

---

## 必做项

- 对前端有实际影响的改动，至少执行 `npm.cmd run build` 验证。
- 能落入现有测试覆盖范围的行为，优先补或更新 `npm run test:unit`。
- 代码结构保持 Vue 3 Composition API 与 `script setup` 风格一致。
- 路由页面继续放在 `views/`，共享 UI 继续放在 `components/`。
- store 状态、props、共享数据结构都要补齐 TypeScript 类型。
- 修改现有模块时，尽量保持 Element Plus 和 SCSS 的既有接入方式不变。

---

## 校验要求

- 当前仓库已经具备基础 `Vitest + Vue Test Utils` 能力，并存在 `npm run test:unit` 入口，但并不代表所有前端改动都已具备完整测试覆盖。
- 代码改动的最低验证标准是：
  - `npm.cmd run build`
  - 能落入现有测试覆盖范围的行为，优先补或更新 `npm run test:unit`
  - 当 UI 或路由发生变化时，做一次针对性的手工冒烟检查
- 如果任务引入了复杂逻辑而现有测试覆盖无法支撑，应优先补可复用测试，而不是临时发明一套零散校验。

---

## 代码评审检查项

- 改动是否保持了 Vue 3 + TypeScript 的整体结构一致性？
- props、emits、store state 是否有明确类型？
- 路由改动是否同步反映到菜单或布局行为？
- Pinia 持久化状态是否仍然是“有意且最小化”的？
- 是否把无关重构混进了同一次任务？
- 如果属于前端行为变化任务，是否已按 `frontend-docs-first-tdd` 完成需求确认？
- 是否保留了测试依据或验证依据？
- 是否说明本次任务是“先失败，再通过”，若没有，是否存在已说明的降级理由？
