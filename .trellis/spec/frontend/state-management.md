# 状态管理规范

> 这个项目中的状态应该如何管理。

---

## 总览

全局状态使用 `Pinia`。持久化能力通过 `src/stores/index.ts` 中集成的
`pinia-plugin-persistedstate` 提供。

---

## 状态分类

- 组件本地状态：
  菜单折叠、刷新开关、弹窗显示、表单草稿等只在视图中使用的状态，放在
  `script setup` 里的 `ref` / `computed` 中。
- 全局应用状态：
  用户信息、权限、认证状态、跨多个路由共享的数据，放在
  `src/stores/modules/` 下的 Pinia 模块里。
- URL 状态：
  由路由驱动的状态交给 `vue-router`，不要再重复存进 store。
- 服务端状态：
  当前没有专门的缓存库，按复用范围决定放在页面还是 store action 中。

---

## 何时使用全局状态

- 当多个路由或多个布局区域都需要该状态时，提升到 Pinia。
- 当页面刷新后仍需保留该状态时，提升到 Pinia。
- 纯临时的 UI 开关不要随便放进 Pinia，除非它需要协调多个相隔较远的组件。

---

## 服务端状态处理

- 异步逻辑优先放在 Pinia action 或页面级函数中处理。
- store 对外暴露的应该是稳定、带类型的状态，而不是把未处理完成的 Promise
  直接给模板使用。
- 需要持久化时，只通过 `persist.pick` 精确白名单保留必要字段。

---

## 常见错误

- 不要把所有页面变量都塞进全局 store。
- 不要默认持久化敏感状态或生命周期很短的状态。
- 不要用 `any` 绕过 store 类型约束。
- 修改 `src/stores/modules/user.ts` 时，除非任务明确要求调整认证存储策略，
  否则保持现有 store id 和持久化行为不变。
